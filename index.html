Web Basics

Understanding how the web works is fundamental to web security. Let's explore the core concepts.

Client-Server Model

The web operates on a client-server architecture where clients (browsers) request resources from servers.

How it works:

  • Client: Web browser (Chrome, Firefox, etc.) that initiates requests
  • Server: Remote computer that processes requests and sends responses
  • Request-Response Cycle: Client sends HTTP request → Server processes → Server sends HTTP response

Frontend (Client-Side)

The frontend is what users interact with directly in their browsers.

HTML

Structures web content with elements like forms, inputs, and semantic tags.

<form action="/submit" method="post"> <input type="text" name="username"> <input type="password" name="password"> <input type="submit" value="Login"> </form>

JavaScript

Adds interactivity, manipulates DOM, and handles events.

document.getElementById("myButton") .addEventListener("click", function() { alert("Button clicked!"); });

Cookies & Sessions

Cookies store data on the client, while sessions store data on the server.

// Set a cookie document.cookie = "username=John Doe; expires=...";

Backend (Server-Side)

The backend handles business logic, data processing, and response generation.

Sessions & Authentication

Maintain user state across requests and verify user identities.

// PHP session example session_start(); $_SESSION['user_id'] = 123; $_SESSION['logged_in'] = true;

Databases

Store and retrieve application data.

  • SQL: Structured, relational databases (MySQL, PostgreSQL)
  • NoSQL: Flexible, non-relational databases (MongoDB, Redis)

APIs

Application Programming Interfaces allow communication between different software systems.

// Example REST API endpoint GET /api/users/123 POST /api/users PUT /api/users/123 DELETE /api/users/123

HTTP/HTTPS Deep Dive

HTTP is the foundation of data communication on the web. HTTPS adds security through encryption.

HTTP Methods

HTTP methods indicate the desired action to be performed on a resource.

Method Description Idempotent Safe Request Body Response Body
GET Retrieve a resource Yes Yes Optional Yes
POST Submit data to be processed No No Yes Yes
PUT Update a resource Yes No Yes Usually
DELETE Delete a resource Yes No Optional Usually
HEAD Get headers only Yes Yes Optional No

HTTP Status Codes

Status codes indicate the result of the HTTP request.

2xx Success

  • 200 OK: Request succeeded
  • 201 Created: Resource created
  • 204 No Content: Success but no content

3xx Redirection

  • 301 Moved Permanently: Resource moved permanently
  • 302 Found: Resource temporarily moved
  • 304 Not Modified: Use cached version

4xx Client Errors

  • 400 Bad Request: Malformed request
  • 401 Unauthorized: Authentication needed
  • 403 Forbidden: No permission
  • 404 Not Found: Resource doesn't exist

5xx Server Errors

  • 500 Internal Server Error: Generic server error
  • 502 Bad Gateway: Invalid response from upstream
  • 503 Service Unavailable: Server overloaded

HTTP Headers

Headers provide additional information about the request or response.

GET /index.html HTTP/1.1 Host: www.example.com User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) Accept: text/html,application/xhtml+xml Accept-Language: en-US,en;q=0.5 Accept-Encoding: gzip, deflate Connection: keep-alive Cookie: sessionid=abc123 Referer: https://www.google.com/ Content-Type: application/x-www-form-urlencoded

Important Headers:

  • Host: Specifies the domain name of the server
  • User-Agent: Identifies the client software
  • Cookie: Contains stored cookies for the domain
  • Referer: Indicates the page that linked to the requested resource
  • Content-Type: Specifies the media type of the resource

HTTPS & TLS

HTTPS adds encryption to HTTP using TLS (Transport Layer Security) to secure communications.

TCP vs UDP

These are the two main transport layer protocols with different characteristics.

TCP (Transmission Control Protocol)

  • Connection-oriented
  • Reliable with error checking
  • Sequencing of data packets
  • Flow control
  • Slower but more reliable
  • Used by HTTP, HTTPS, FTP, SMTP

UDP (User Datagram Protocol)

  • Connectionless
  • No error checking
  • No sequencing of packets
  • No flow control
  • Faster but less reliable
  • Used by DNS, VoIP, streaming

IP Addressing and Ports

IP addresses identify devices on a network, while ports identify specific services on a device.

Common Ports:

  • 20/21: FTP (File Transfer Protocol)
  • 22: SSH (Secure Shell)
  • 25: SMTP (Simple Mail Transfer Protocol)
  • 53: DNS (Domain Name System)
  • 80: HTTP (Hypertext Transfer Protocol)
  • 443: HTTPS (HTTP Secure)
  • 3306: MySQL Database
  • 3389: RDP (Remote Desktop Protocol)

NAT & Firewalls

Network security mechanisms that control traffic flow.

NAT (Network Address Translation)

Allows multiple devices on a private network to share a single public IP address.

  • Conserves public IP addresses
  • Adds a layer of security by hiding internal IPs
  • Can cause issues with some protocols

Firewalls

Network security system that monitors and controls incoming and outgoing traffic.

  • Packet filtering: Allows or blocks packets based on rules
  • Stateful inspection: Tracks active connections
  • Proxy service: Intercepts all messages
  • Next-generation: Application-level filtering

File Permissions

Linux uses a permission system to control access to files and directories.

$ ls -la

-rwxr-xr-x 1 user group 2048 Jan 1 12:34 script.sh

drwxr-xr-x 2 user group 4096 Jan 1 12:34 directory

lrwxrwxrwx 1 user group 27 Jan 1 12:34 symlink -> /path/to/target

Permission Breakdown:

  • User (u): Owner permissions (rwx)
  • Group (g): Group permissions (r-x)
  • Other (o): Everyone else permissions (r-x)
  • Read (r): Permission to read file/list directory (4)
  • Write (w): Permission to modify file/add to directory (2)
  • Execute (x): Permission to execute file/enter directory (1)

Web Security Principles

Understanding and implementing security principles is essential for protecting web applications.

Input Validation

All user input should be treated as untrusted and validated before processing.

Client-Side Validation

  • Provides immediate feedback to users
  • Improves user experience
  • Easily bypassed - never rely on it for security
  • Implemented with HTML5 attributes and JavaScript

Server-Side Validation

  • Essential security measure
  • Validates data before processing
  • Protects against malicious input
  • Should use allowlists rather than denylists

Common Validation Pitfalls

  • Using denylists instead of allowlists
  • Not validating data type, length, and format
  • Not sanitizing data before output
  • Not validating data on the server side

Output Encoding (XSS Protection)

Encoding data before output prevents it from being interpreted as code by browsers.

Context-Specific Encoding

  • HTML Context: Convert &, <, >, ", ' to HTML entities
  • HTML Attribute Context: Similar to HTML but with quotes
  • JavaScript Context: Escape for JavaScript strings
  • URL Context: URL encoding with percent-encoding
  • CSS Context: Escape for CSS values
// PHP example: HTML encoding $user_input = "<script>alert('XSS')</script>"; $safe_output = htmlspecialchars($user_input, ENT_QUOTES, 'UTF-8'); echo $safe_output; // Outputs: &lt;script&gt;alert(&#x27;XSS&#x27;)&lt;/script&gt;

Authentication vs. Authorization

These two concepts are fundamental to access control in web applications.

Authentication

Verifying the identity of a user or system.

  • Something you know (password)
  • Something you have (token, phone)
  • Something you are (biometrics)
  • Multi-factor authentication combines these

Authorization

Determining what an authenticated user is allowed to do.

  • Role-Based Access Control (RBAC)
  • Attribute-Based Access Control (ABAC)
  • Permission checks on resources
  • Principle of least privilege

Common Security Issues

  • Weak password policies
  • Plain text password storage
  • Insufficient password hashing
  • Broken authentication logic
  • Insecure direct object references
  • Missing access control checks

HTTPS / TLS Best Practices

Proper implementation of HTTPS is crucial for securing data in transit.

HTTPS Implementation Checklist

  • Use strong certificates from trusted CAs
  • Redirect all HTTP traffic to HTTPS
  • Use HSTS header to enforce HTTPS
  • Implement perfect forward secrecy
  • Use secure cipher suites
  • Keep TLS libraries up to date
  • Use secure cookies (Secure and HttpOnly flags)

Combining Web, Network & OS Security

Real-world security issues often span multiple layers of the technology stack.

SQL Injection

Requires understanding of web applications and databases.

// Vulnerable code $query = "SELECT * FROM users WHERE id = " . $_GET['id']; // Secure code with prepared statements $stmt = $pdo->prepare("SELECT * FROM users WHERE id = :id"); $stmt->execute(['id' => $_GET['id']]);

Path Traversal

Requires understanding of OS file systems.

// Vulnerable code $file = $_GET['file']; readfile("/var/www/files/" . $file); // Secure code $base = "/var/www/files/"; $realBase = realpath($base); $userPath = $base . $_GET['file']; $realUserPath = realpath($userPath); if ($realUserPath === false || strpos($realUserPath, $realBase) !== 0) { // Invalid path die("Invalid file"); }

SSRF (Server-Side Request Forgery)

Requires understanding of networking and web requests.

// Vulnerable code $url = $_GET['url']; $data = file_get_contents($url); // Secure code $allowed_domains = ['example.com', 'api.example.com']; $url = $_GET['url']; $host = parse_url($url, PHP_URL_HOST); if (!in_array($host, $allowed_domains)) { die("Domain not allowed"); }
